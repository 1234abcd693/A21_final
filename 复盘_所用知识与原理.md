# 知识图谱设计工作复盘 —— 所用知识与原理

> 回顾：从《船舶电气设备维护与修理》教材文本出发，经过多轮审阅（产品经理视角 + 航海电气专家视角），设计出可导入 Neo4j 的 v4.0 知识图谱本体。

---

## 一、知识图谱本体设计

### 1.1 实体与关系的抽象能力

**用到的知识**：本体工程（Ontology Engineering）。从非结构化文本中识别出"什么东西"→实体（Entity），"什么关系"→关系（Relation）。

**原理**：实体关系模型（E-R Model）是知识图谱的基石。设计过程是自底向上的——先通读教材提取故障条目，再归纳共性，抽象出 7 类节点和 7 类关系。

### 1.2 图遍历与查询复杂度

**用到的知识**：图论中的路径遍历。Neo4j 的 `MATCH path = (a)-[:REL*0..]->(b)` 支持变长路径。

**原理**：知识图谱的查询效率取决于关系边的方向设计。好的设计让常见查询只需 1~2 跳。例如"电动机不能起动的所有原因"：`Symptom → CAUSED_BY → Cause → FIXED_BY → Step`，链式遍历无回溯。

### 1.3 属性在节点 vs 在关系上的抉择

**用到的知识**：属性图模型（Property Graph Model）。Neo4j 中节点和关系都可以携带属性。

**关键决策**：`priority` 放 CAUSED_BY 关系边上，而非 Cause 节点上。同一 Cause 可被多 Symptom 共享，不同 Symptom 下优先级不同，关系边存不同 priority，互不冲突。

## 二、图数据库（Neo4j）专项知识

### 2.1 MERGE vs CREATE

`CREATE` 每次执行都新建节点；`MERGE` 先查找匹配，找到则复用，找不到则创建，保证幂等性。

### 2.2 变量长度路径

`*0..` 表示 0 到任意深度。用于 Equipment 层级下钻。

### 2.3 UTF-8 BOM 与 Windows 编码陷阱

Windows 中文版默认编码是 GBK。UTF-8 文件不带 BOM（`\xEF\xBB\xBF`），Windows 记事本会按 GBK 解码，导致中文字符乱码。

## 三、产品管理 / 审阅方法论

本次工作经历了 5 轮审阅：

| 轮次 | 角色 | 发现的核心问题 |
|------|------|-------------|
| 第1轮 | 基础审阅 | 缺溯源属性、缺 step_order |
| 第2轮 | 产品经理 | 缺 SparePart/Tool/Precaution、Equipment 缺层级 |
| 第3轮 | 严格审阅 | 缺 Markdown 分隔行、Emoji 风险、G-05 占位符、UID 脱节 |
| 第4轮 | 深度审阅 | Cause 共享约束过严、step_order+NEXT_STEP 冗余 |
| 第5轮 | 最终审阅 | RepairProcedure 冗余→删除、Tool 缺可替代性、Precaution 缺后果 |

每次审阅用不同视角发现不同层次的问题。多角色模拟审阅是质量控制的关键手段。

### 优先级分级（P0/P1/P2）

- **P0**：阻塞性问题（无法导入、数据损坏、编码风险）
- **P1**：设计完整性问题
- **P2**：增强项

## 四、航海电气设备维修领域知识

### 4.1 故障诊断的"先易后难"原则

船舶设备维修遵循"由易到难、由简到繁、由表及里"的排查顺序。这直接体现在 Cause 的 `priority` 排序上——目视检查排在万用表测量前面，万用表测量排在拆机前面。

### 4.2 维修流程的多步性

实际维修不是单步操作。"定子绕组接地"需要：测量绝缘→定位故障点→修复。三者有严格的先后依赖，必须体现为有序步骤。

### 4.3 船舶设备的安全特殊性

船舶是封闭空间，高压设备密集，维修安全要求极高。Precaution 节点表达三层信息：严重等级（forbidden/warning/info）、违反后果（如"高压电击致死"）、后果类型（injury/equipment_damage/data_error）。

### 4.4 船舶工具的可替代性

远洋船舶工具储备有限。工具节点需要 `alternatives` 和 `irreplaceable_for`，辅助船员做工具替代决策。

## 五、数据工程知识

### 5.1 溯源体系（Provenance）

RAG 系统的答案必须可追溯。每个知识节点携带 `source_doc`、`source_page`、`source_author`、`source_publisher`，确保系统生成的答案能定位到教材原文。这是对抗大模型幻觉的关键机制。

### 5.2 幂等导入

`MERGE` + `uid` 唯一键确保脚本可以多次执行而不产生重复数据。

### 5.3 分离文档 ID 与数据库 UID

设计文档中的 `C-01`、`T-02` 是给人读的引用号。实际导入使用 `S_CONT_04`、`C_CONT_04a` 格式的规范化 UID。

## 六、Cypher 脚本编写

### 6.1 FOREACH 批量操作 + 动态 UID 拼接

当多个 Cause 共享相同的修复流程结构时，FOREACH 可避免重复代码。

```
FOREACH (c IN [c_voltage_low, c_c01b, c_coil_short, ...] |
  MERGE (proc:RepairProcedure {uid: 'PROC_' + c.uid})
  ...
)
```

### 6.2 防范重复关系创建

在执行关系创建前用 `WHERE NOT EXISTS((st)-[:HAS_PRECAUTION]->())` 检查是否已存在。

### 6.3 大规模脚本的编写策略

- 按设备分组
- 共享 Cause 前置
- 批量收尾
- 验证查询收尾

### 6.4 Cypher 脚本的常见错误

| 错误 | 后果 | 修正 |
|------|------|------|
| MERGE 后不带 SET 就创建关系 | 节点只有 uid | 先 MERGE 节点再 SET |
| FOREACH 内 MATCH 引用外部变量 | 变量作用域丢失 | FOREACH 内只用 MERGE/CREATE/SET |
| 关系边忘记加属性 | 无法排序 | `MERGE (p)-[:HAS_STEP {sequence:1}]->(s)` |
| uid 含特殊字符 | 查询需转义 | uid 只用 `[A-Z0-9_]` |
