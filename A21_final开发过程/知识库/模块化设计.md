# 模块化设计

## 一句话解释

把程序拆成**独立的小块**，每块只干一件事，换一块不影响其他。

## 类比：电脑的 USB 接口

你换鼠标不需要拆机箱，插上 USB 就行。因为电脑和鼠标之间有一个**标准接口（USB）**。

坏比：
- ❌ 鼠标线直接焊在主板上 → 换鼠标要用电烙铁
- ✅ 鼠标通过 USB 连接 → 拔下来插新的就行

模块化 = 所有模块之间都用"USB 接口"连接。

## 在我们的项目里怎么体现

### 反馈调优模块

```
任何优化算法必须实现 optimize() 方法：
  输入：反馈数据列表
  输出：最优参数字典

当前用：GridSearchOptimizer（网格搜索）
以后换：BayesianOptimizer（贝叶斯优化）
→ 只需改一行代码，其他模块完全不受影响
```

### RAG 检索模块

```
任何检索器必须实现 search() 方法：
  输入：用户问题字符串
  输出：文档片段列表（带分数）

当前用：HybridRetriever（三路融合）
以后换：任何新算法
→ 只需改一行代码
```

### 知识抽取模块

```
任何抽取器必须实现 extract() 方法：
  输入：纯文本
  输出：候选实体列表

当前用：RegexExtractor + FewShotExtractor
以后换：微调的 NER 模型
→ 只需改一行代码
```

## 怎么做：抽象基类

Python 的 `ABC`（Abstract Base Class）就是"接口规范"：

```python
from abc import ABC, abstractmethod

# 第一步：定义接口（"所有优化器必须长这样"）
class BaseOptimizer(ABC):
    @abstractmethod
    def optimize(self, feedbacks: list[dict]) -> dict:
        """必须实现这个方法"""
        pass

# 第二步：实现接口
class GridSearchOptimizer(BaseOptimizer):
    def optimize(self, feedbacks):
        # 具体实现...
        return best_params

# 第三步：使用接口（不关心具体是哪个实现）
def run_optimization(optimizer: BaseOptimizer):
    feedbacks = load_feedbacks()
    params = optimizer.optimize(feedbacks)   # 不管 optimizer 是谁，都调 optimize()
    save_params(params)
```

## 为什么重要？

| 没有模块化 | 有模块化 |
|-----------|---------|
| 换算法 → 到处改代码，可能引入 bug | 换算法 → 写一个新类，改一行引用 |
| 两个人在不同模块工作 → 互相踩脚 | 两个人在不同模块 → 只要接口不变，互不影响 |
| 出了问题不知道在哪 | 每个模块独立测试，快速定位 |
| 代码像一碗意大利面 | 代码像乐高积木 |

## 我们项目里所有需要模块化的地方

| 模块 | 接口方法 | 当前实现 | 未来可能换 |
|------|---------|---------|-----------|
| 优化算法 | `optimize(feedbacks) → params` | GridSearch | Bayesian |
| 检索器 | `search(query) → docs` | HybridRetriever | 改进的融合算法 |
| 知识抽取 | `extract(text) → entities` | Regex+FewShot | 微调 NER |
| Embedding | `embed(text) → vector` | MiniLM | bge-large-zh |
| Reranker | `rerank(docs) → docs` | 待定 | 换模型 |
| LLM 后端 | `generate(prompt) → tokens` | llama.cpp | 换模型/换引擎 |
