# 向量数据库同步原理

## 一句话解释

向量数据库 = 一个文件夹。同步 = 拷贝这个文件夹。但有个前提：两边用的 embedding 模型必须一样。

## Chroma 的数据存在哪里？

```
data/chroma_db/
├── chroma.sqlite3       ← 元数据（文档名、分块信息、创建时间等）
└── 向量索引文件          ← embedding 向量（768 维 × 数量）
```

`chroma.sqlite3` 是普通 SQLite 数据库，存的是"这条向量属于哪个文档、第几页"。向量索引文件存的是实际的高维向量。

**两者缺一不可**——元数据告诉你"这是什么"，向量索引让相似度搜索能工作。

## 同步流程

### 导出

```
1. 确保没有正在写入 Chroma 的进程（问答时在读取，不会写；知识抽取入库时会写）
2. 直接复制整个 chroma_db/ 文件夹
3. 塞进 zip 里
```

### 导入

```
1. 解压 zip → 得到 chroma_db/
2. 读取 sync_info.json 里的 embedding_model 字段
3. 检查本地 embedding 模型是否一致：
   - 一致 → 合并数据（新文档插入，重复跳过）
   - 不一致 → 警告用户："两边用的 embedding 模型不同，向量需要重新计算"
            → 用户选择：取消 / 强制导入（全部重新 embedding，耗时较长）
4. 合并完成后，重启 Chroma 客户端（重新加载索引）
```

## 为什么模型必须一致？

类比：两个人用不同的尺子量同一张桌子。

```
设备 A 用 MiniLM (384维) 算出的向量:
  "发动机漏油" → [0.23, 0.87, -0.15, ..., 0.41] (384个数字)

设备 B 用 bge-base-zh (768维) 算出的向量:
  "发动机漏油" → [0.31, 0.72, -0.09, ..., 0.55, ...] (768个数字)

如果设备 B 的 Chroma 突然收到 384 维的向量 → 直接报错
如果设备 A 用 MiniLM，设备 B 也用 MiniLM → 正常，向量空间一致
```

**解决方案**：
- `sync_info.json` 记录 `embedding_model`
- 导入时自动比对，不一致就提示

## 如果模型不一致怎么办？

两个选择给用户：

| 选项 | 操作 | 耗时 | 结果 |
|------|------|------|------|
| 取消导入 | 什么也不做 | 0 | 安全，但知识没同步 |
| 强制导入并重新计算 | 逐条新文档重新 embedding | 每条 ~10ms，500 条 ~5 秒 | 知识同步过来，向量可用 |

## 和 Neo4j 同步的区别

| | Neo4j | Chroma |
|------|--------|--------|
| 导出方式 | Cypher 文本 | 复制文件夹 |
| 合并方式 | MERGE 幂等插入 | hash 去重 + 新文档追加 |
| 依赖 | 不需要版本一致 | **embedding 模型必须一致** |
| 速度 | 500 实体 ~1 秒 | 500 条 chunk ~1 秒（不加重新 embedding） |
| 能否增量 | ✅ MERGE 自动处理 | ✅ hash 去重 |

## 为什么同步的是文件夹而不是 SQL 或 API？

向量是浮点数数组，不存在"修改某一条向量"的需求。整个文件夹拷过去最简单、最快、最可靠。500 条 chunk 的 chroma_db/ 大约 5-10 MB，U盘秒传。
