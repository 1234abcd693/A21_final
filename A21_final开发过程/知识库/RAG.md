# RAG（检索增强生成）

## 一句话解释

RAG = 让 AI **先查资料，再回答**，而不是靠记忆瞎编。

## 类比

想象你参加开卷考试 vs 闭卷考试：

| | 闭卷（纯 LLM） | 开卷（RAG） |
|------|-----------|---------|
| 知识来源 | 训练时记住的东西，可能忘了或记错了 | 现场翻书查资料，准确 |
| 是否可追溯 | 不知道答案从哪来的 | 每个答案都标注了引用来源 |
| 新知识 | 不知道（训练截止后的事） | 随时加新资料就能查到 |

## 工作流程

```
用户提问："发动机无法启动怎么办？"
    │
    ▼
┌──────────────┐
│  检索阶段     │  ← 从知识库里找出相关的内容
│  (Retrieval) │     关键词匹配 + 语义搜索
└──────┬───────┘
       │ 相关文档片段（比如维修手册的第3页第5段）
       ▼
┌──────────────┐
│  增强阶段     │  ← 把找到的资料和用户问题拼在一起
│ (Augmented)  │     "请根据以下资料回答问题：{资料} 问题：{用户提问}"
└──────┬───────┘
       │ 构造好的 Prompt
       ▼
┌──────────────┐
│  生成阶段     │  ← LLM 读资料，组织语言输出答案
│ (Generation) │     不是凭记忆回答，是"阅读理解+总结"
└──────────────┘
       │
       ▼
  带引用的结构化答案
```

## 为什么我们这个项目必须用 RAG？

1. 船舶故障诊断是**高度专业化的知识**，通用 LLM 的训练数据里很少有
2. 我们用的是 **1.5B 小模型**，它记不住太多知识
3. 赛题要求**答案可追溯、有依据**——RAG 天然带引用
4. 新故障案例可以随时加入知识库，模型不需要重新训练

## 常见误区

- ❌ "RAG 就是给 LLM 加点上下文" → RAG 的难点在**检索**，检索不准一切都白搭
- ❌ "大模型不需要 RAG" → 大模型也会幻觉，专业领域 RAG 必不可少
- ❌ "RAG 只能搜文本" → 可以搜知识图谱、数据库、API 等等
