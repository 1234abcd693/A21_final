# Prompt 缓存

## 一句话解释

LLM 处理 prompt 时，**不变的部分只算一次，变化的部分每次重新算**。

## 类比

老师每次上课开头都念一遍"同学们好，今天我们学习第X章"。如果这段话每次都重新想一遍就很慢。

更好的做法：这段话印在教案第一页，**翻开就看到了**，直接跳到新内容。

Prompt 缓存就是这个道理——系统指令是"印好的教案"，直接复用，只处理新的问题+资料。

## 为什么重要？

RAG 系统的 prompt 结构：

```
┌────────────────────────────────┐
│ 系统指令（每次一样）             │  ← 这部分可以缓存！
│ "你是船舶维修专家。请根据以下    │
│  资料回答问题。如果资料中没有    │
│  相关信息，回答未找到..."        │
├────────────────────────────────┤
│ 检索结果（每次不同）             │  ← 每次重新处理
│ [1] 发动机无法启动时...         │
│ [2] 燃油系统常见故障...         │
├────────────────────────────────┤
│ 用户问题（每次不同）             │  ← 每次重新处理
│ "发动机打不着火怎么办？"         │
└────────────────────────────────┘
```

没有缓存：每次处理 100% 的 token（~500 token）  
有缓存：只处理变化部分（~250 token），**节省一半时间**

## llama.cpp 怎么开启 Prompt 缓存

llama-server 模式自带缓存，通过以下方式：

```bash
llama-server -m model.gguf \
  --cache-type-k q8_0 \     # KV cache 用 8-bit 量化，省内存
  --cache-reuse 256         # 复用最近 256 个 token 的缓存
```

第一次调用：处理全部 prompt → 缓存系统指令部分的 KV Cache  
第二次调用：系统指令部分"命中缓存"，只处理新内容

## 缓存什么？KV Cache 是什么？

LLM 处理 prompt 的过程：
1. 读第 1 个 token → 计算出 K1, V1（键值对，中间结果）
2. 读第 2 个 token → 参考 K1,V1 → 算出 K2,V2
3. ...
4. 全部读完后，基于所有 KV 对，开始生成第 1 个新 token

KV Cache 就是这些中间结果（K1,V1, K2,V2...）。缓存后，下次相同前缀的 token 直接复用 KV，不需要重新计算。

## 效果

| 场景 | 无缓存 TTFT | 有缓存 TTFT |
|------|-----------|-----------|
| 首轮对话 | ~3.5s | ~3.5s（首次无缓存可命中） |
| 后续轮次 | ~3.5s | ~1.5-2.0s ✅ |
| 切换话题 | ~3.5s | ~2.0-2.5s（系统指令仍缓存） |
