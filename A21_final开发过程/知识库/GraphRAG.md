# GraphRAG 是什么，我们为什么不需要

## 一句话解释

GraphRAG = 用知识图谱做 RAG 检索。我们已经在做了，不需要额外的框架。

## 两个"GraphRAG"——容易混淆

### 1. Microsoft GraphRAG（框架）

微软 2024 年开源的一个**完整框架**。工作流程：

```
文档 → LLM 提取实体和关系 → 构建知识图谱 → 
                                          → 社区检测（找紧密关联的群体）
                                          → LLM 给每个社区生成摘要
                                          → 检索时：图遍历 + 向量搜索 + 社区摘要 三路融合
```

**问题**：
- 索引阶段需要**大量 LLM 调用**（提取实体、生成摘要），1.5B 小模型撑不住
- 社区检测算法在 CPU 上跑几千个实体很慢
- 是为**百万级实体**的大规模非结构化语料设计的
- 每次加新文档都要重新索引

### 2. Neo4j GenAI 插件

你 Neo4j 目录里已经有的 `neo4j-genai-plugin-2026.02.3.jar`。

功能：在 Neo4j 里直接存 embedding 向量，用 Cypher 做向量相似度搜索。

也就是——把 Chroma 的工作挪到 Neo4j 里做。

## 我们的方案对比

| | 我们的方案 | Microsoft GraphRAG | Neo4j GenAI 插件 |
|------|-----------|-------------------|-----------------|
| 图检索 | ✅ Cypher 查询 | ✅ 社区+遍历 | ✅ Cypher（自带） |
| 向量检索 | ✅ Chroma | ✅ 内置 | ✅ Neo4j 内 |
| 关键词检索 | ✅ BM25 | ❌ 没有 | ❌ 没有 |
| 索引成本 | 低（embedding 一次） | **高**（LLM 多次调用） | 低 |
| 依赖 | Chroma（已用） | 额外框架 + 大量 LLM 调用 | 额外插件 |
| 小模型友好 | ✅ | ❌（需要强 LLM 做提取） | ✅ |
| 500 实体规模 | ✅ 刚好 | ❌ 过度设计 | ✅ |

## 结论

**我们不需要引入任何"GraphRAG"框架或插件。**

我们的方案本身就是 GraphRAG 的思想——图检索 + 语义检索融合。区别在于我们**手工设计了知识图谱**（而不是让 LLM 自动提取），并且用 **BM25 补齐了关键词检索**（这是两个 GraphRAG 方案都没有的）。

对于 500 实体、80 关系的规模，手工设计的知识图谱 + 三路融合远比框架自动建图更精确。
