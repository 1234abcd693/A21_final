# 数据库原理（开发者视角）

## 一句话解释

数据库 = 一个**独立运行的程序**，你的代码通过网络和它对话。不是在代码里 `import` 一个库。

## 你的理解完全正确

```
开发阶段:
  你在电脑上安装 Neo4j → 启动 → 导入 import.cypher
  → 你的 Python 代码通过 bolt://localhost:7687 连接它 → 查询、写入

分发阶段:
  用户电脑上没有 Neo4j
  → 程序包里自带 Neo4j 绿色版 + 初始数据
  → 你的程序自动启动 Neo4j → 自动导入数据 → 用户代码连接本地的 Neo4j
  → 用户的问答、反馈全都存在他自己的本地数据库里
```

**每个用户的 Neo4j 是独立的**，互不影响。就像每个用户自己装了一个 Excel，打开的是自己的表格文件。

## 数据库不是什么

很多初学者以为数据库是这样：
```
❌ 误解: 数据库 = 一个文件，用 Python 打开它就能读写
❌ 误解: 数据库 = import 进来的库，数据存在内存里
```

实际是这样：
```
✅ 数据库 = 一个独立的后台服务，监听端口，等待连接
✅ 你的代码 = 客户端，通过"连接"发送"查询语句"
✅ 数据 = 存在数据库自己的文件里（Neo4j 的 data/ 目录）
```

## 类比

| | 餐厅 | 数据库 |
|------|------|--------|
| 厨师 | Neo4j 服务进程 | 独立运行的 neo4j.exe |
| 菜单 | Cypher 查询语言 | `MATCH (n) RETURN n` |
| 点菜 | 发送查询 | Python 发送 Cypher |
| 厨房 | 数据库存储 | `data/databases/neo4j/` 目录 |

你不是直接进厨房拿菜——你通过服务员（bolt://localhost:7687）点菜，厨师（Neo4j 进程）做好后端出来。

## 我们项目的数据库生命周期

```
1. 开发阶段（你）
   ├── 安装 Neo4j Community 版
   ├── 启动服务
   ├── 导入 import.cypher（初始化数据）
   ├── 开发 Python 代码 → 验证查询 → 调整数据
   └── 最后：导出一个"干净"的初始化脚本 → import.cypher (最终版)

2. 打包阶段
   ├── 把 Neo4j 绿色版 放入 resources/
   ├── 把 import.cypher 放入 resources/
   └── 打包成 Setup.exe

3. 用户首次启动
   ├── 程序自动启动 Neo4j（首次启动创建空数据库）
   ├── 程序自动执行 import.cypher（导入初始知识）
   ├── 现在用户有了一个包含 500+ 实体的知识图谱
   └── 用户的每次问答 → 查他自己的 Neo4j → 结果是他个人的

4. 用户日常使用
   ├── 用户上传文档 → 知识抽取 → 写入 Neo4j（添加新实体）
   ├── 用户问答 → 查 Neo4j（含他自己添加的知识）
   ├── 用户导出 U盘 → 导出他的 Neo4j 数据（Cypher）
   └── 全部存在用户本机，不上传到任何服务器
```

## 关键理解

- 你开发的**不是**一个公共数据库的客户端
- 你开发的**是**一个自带数据库的桌面软件
- 每个用户运行的是**自己独立的 Neo4j 实例**
- 数据和程序都在用户本机，完全离线
