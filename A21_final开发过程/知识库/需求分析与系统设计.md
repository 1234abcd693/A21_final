# 需求分析与系统设计

## 一句话解释

拿到一个项目不要直接写代码。先搞清楚**谁要用、用来干什么、怎么做**，再动手。

## 三步法

### 第一步：需求分析 — "用户要什么？"

```
读赛题文档 → 找出所有"必须做到"的事情
     ↓
整理成需求列表，每一条都是可验证的：
  ✅ "问答响应 ≤3秒"      ← 好，可以测
  ❌ "系统要快"            ← 差，不知道多快算快
```

### 第二步：功能分解 — "系统要能干什么？"

每个需求拆成具体的功能点：

```
需求: "支持故障诊断问答"
  ├── 功能: 用户输入问题
  ├── 功能: 系统检索知识库
  ├── 功能: LLM 生成回答
  ├── 功能: 流式展示答案
  └── 功能: 显示引用来源
```

### 第三步：模块设计 — "哪个模块负责什么？"

每个功能点分配给一个具体模块：

```
功能: LLM 生成回答 → RAG 层 / generator.py
功能: 流式展示答案 → 前端 MessageBubble.vue + 后端 StreamingResponse
```

## 为什么这样做事？

| 跳步直接写代码 | 先分析再设计 |
|---------------|------------|
| 写到一半发现漏了功能 | 需求追踪矩阵保证全覆盖 |
| 不知道这个功能该放哪个文件 | 模块职责清晰，不会放错 |
| 队友不知道你做到了哪一步 | 状态一目了然 |
