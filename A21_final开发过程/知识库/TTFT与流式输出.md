# TTFT 与流式输出

## TTFT 是什么

**TTFT = Time To First Token = 首个 token 生成时间**

从用户点击发送，到屏幕上出现第一个字的时间间隔。

## 为什么要关心 TTFT？

用户等待的耐心是有限的。研究表明：
- **< 1 秒**：感觉是即时的
- **1-3 秒**：可以接受，用户在等待
- **3-10 秒**：开始不耐烦
- **> 10 秒**：用户可能关掉窗口

我们的赛题要求："单轮问答响应时间不超过 3 秒"。

## TTFT 包含什么

```
用户点击发送
    │
    ├─ 检索阶段（~0.5s）
    │   · 关键词提取、实体识别
    │   · 知识图谱查询
    │   · 向量检索 + BM25 检索
    │   · 结果融合排序
    │
    ├─ Prompt 处理阶段（~1.5s，最大头）
    │   · 把检索结果和用户问题拼成 prompt
    │   · LLM 逐 token 读取整个 prompt
    │   · 这是最耗时的步骤！
    │
    └─ 生成首 token（~0.05s）
        · LLM 开始输出第一个字
        · 到这一步，用户看到第一个字符 → 计时结束
```

## 流式输出是什么

| 方式 | 体验 |
|------|------|
| 非流式 | "..." 转圈转了 10 秒，然后一次性输出整段回答 |
| **流式（SSE）** | 第 2 秒出现第一个字，然后逐字逐字打出来 |

流式输出**不缩短总时间**，但用户**体感好很多**——看到字在跳动就知道系统在干活。

## 我们的策略

1. **Prompt 缓存**：系统指令永久缓存，每次只处理新内容（省 1-2 秒）
2. **小模型 1.5B**：Prompt 处理快
3. **SSE 流式**：让用户立刻看到第一个字
4. **压缩上下文**：控制 prompt 长度 ≤400 token

有了这些组合，TTFT 可以控制在 **2 秒左右**，满足 3 秒要求。

## 相关概念

- **Token**：LLM 处理文本的最小单位。一个中文字 ≈ 1.5-2 个 token
- **KV Cache**：模型在处理 prompt 时产生的中间结果，缓存后下次可以复用
- **SSE**（Server-Sent Events）：服务端主动推送数据给前端的协议，流式输出常用
