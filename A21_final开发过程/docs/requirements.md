# A21 船舶装备故障诊断智能问答系统 — 需求规格

> 中国大学生服务外包创新创业大赛 A21 赛题  
> 命题单位：大连海事大学

---

## 一、赛题硬性指标

### 1.1 功能要求

| 编号 | 要求 | 说明 |
|------|------|------|
| F-01 | 知识覆盖度 | 至少支持船舶发动机、液压系统、电气设备等 **3 类关键装备** 的常见故障诊断 |
| F-02 | 问答交互 | 支持 **文本输入** 与 **关键词检索** 两种交互方式 |
| F-03 | 历史对话 | 支持历史对话管理，答案可导出 |
| F-04 | 知识可视化 | 知识图谱可视化界面，展示故障-原因-解决方案关联关系 |
| F-05 | 可扩展性 | 架构支持新增装备类型与故障案例的快速导入 |
| F-06 | 离线运行 | 完全离线环境可正常提供问答服务 |

### 1.2 性能指标

| 编号 | 指标 | 要求 |
|------|------|------|
| P-01 | 知识图谱实体数 | ≥ 500 |
| P-02 | 知识图谱关系数 | ≥ 80 |
| P-03 | 语义+图谱融合召回率 | ≥ 90% |
| P-04 | 最终解答准确率（专家评估） | ≥ 80% |
| P-05 | Top1 答案精确匹配准确率 | ≥ 75% |
| P-06 | 单轮问答响应时间 | ≤ 3 秒 |
| P-07 | 系统内存占用 | ≤ 8 GB |
| P-08 | 系统启动时间 | ≤ 30 秒 |
| P-09 | 模型参数规模 | ≤ 7B，仅限开源模型 |

### 1.3 技术约束

| 编号 | 约束 |
|------|------|
| T-01 | 知识图谱：Neo4j 或 GraphDB（二选一） |
| T-02 | NLP 框架：Python + LangChain |
| T-03 | 前端：推荐 Streamlit 或 Flask+HTML，可用其他框架 |
| T-04 | 模型仅用开源，参数 ≤ 7B |
| T-05 | 交付 Windows 可执行程序包（无需安装依赖） |

### 1.4 提交材料

- 项目概要介绍、PPT、详细方案
- 演示视频（3 分钟）
- 知识图谱设计文档（含本体结构、实体关系定义）
- 源代码调用关系图
- 可执行程序包（Windows）
- 测试报告（含 50 题测试集及系统响应）
- 团队自愿补充材料

## 二、用户反馈闭环

### 2.1 需求描述

用户对每一次系统回答可以进行 **点赞（👍）** 或 **点踩（👎）** 评价，反馈数据发送到后端后，系统自动对 RAG 检索链路中的某个环节进行参数调优，持续提升回答质量。

### 2.2 数据流

```
用户提问 → RAG 检索 → LLM 生成 → 回答展示 → 用户评价(👍/👎)
                                                    ↓
                                          反馈数据持久化(SQLite)
                                                    ↓
                                          触发参数优化算法
                                                    ↓
                                          更新 RAG 层参数
```

### 2.3 待设计决策

| 问题   | 选项                                          |
| ---- | ------------------------------------------- |
| 调优对象 | 检索权重(α/β)、重排序阈值、embedding 模型选择、top-k 数量……待定 |
| 调优算法 | 网格搜索、贝叶斯优化、遗传算法、强化学习……待讨论                   |
| 触发方式 | 手动触发、"积累 N 条反馈后自动触发"、实时在线更新                 |
| 反馈存储 | SQLite（本地离线）                                |
| 额外信息 | 是否需要用户填写文字评价？可选                             |

### 2.4 约束

- 调优过程不能阻塞正常问答请求
- 调优参数需可持久化，重启后保留
- 提供"恢复默认参数"能力

## 三、U 盘跨设备数据同步

### 3.1 需求描述

支持通过 U 盘在不同设备间迁移和合并系统数据。典型场景：船 A 积累了一批故障案例和反馈数据，船员将数据导出到 U 盘，插入船 B 的电脑，导入并合并数据。

### 3.2 需同步的数据

| 数据 | 存储位置 | 说明 |
|------|----------|------|
| 知识图谱 | Neo4j | 实体、关系、属性 |
| 向量库 | Chroma | 文档切片 + embedding |
| 用户反馈 | SQLite | 点赞/点踩记录、文字评价 |
| 调优参数 | 配置文件 | RAG 层权重等 |
| 对话历史 | 待定 | 是否需要？ |

### 3.3 操作流程

```
导出: 系统 → 打包(zip) → 写入 U 盘
导入: U 盘 → 解包 → 停止本地服务 → 覆盖/合并数据 → 重启服务
```

### 3.4 待设计决策

| 问题 | 选项 |
|------|------|
| 打包格式 | 单文件 zip |
| 冲突策略 | 覆盖？合并？保留本地？用户选择？ |
| 安全性 | 导入前校验完整性？是否需要密码？ |
| 操作界面 | 一键按钮 or 多步向导？ |
| Neo4j 热导出 | 是否需要停止 Neo4j 才能复制数据目录？ |
