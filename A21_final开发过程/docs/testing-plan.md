## 二、单元测试（开发阶段）

### 2.1 检索模块

| 用例 | 输入 | 预期 |
|------|------|------|
| BM25-1 | `"发动机无法启动"` | 返回 ≥3 条文档片段，第一条包含"启动" |
| BM25-2 | `"燃油压力 3.2MPa"` | 精确匹配到包含"3.2MPa"的文档 |
| 向量-1 | `"打不着火"` | 语义匹配到"无法启动"相关文档 |
| 向量-2 | `"漏油原因"` | 返回的文档分数 > 0.5 |
| 图检索-1 | `"接触器线圈烧毁"` | 返回 Symptom→Cause→Step 完整链 |
| 图检索-2 | `"不存在的故障"` | 返回空列表 |
| 融合-1 | 三路各 3 条结果 | RRF 去重后 ≤9 条 |
| 融合-2 | 无效参数 α=2.0 | 返回错误码 |

### 2.2 生成模块

| 用例 | 输入 | 预期 |
|------|------|------|
| 生成-1 | 正常 prompt + 检索结果 | 返回带 [1][2] 引用标记的回答 |
| 生成-2 | 空的检索结果 | 返回"未找到相关信息" |
| 生成-3 | 超长 prompt (>2048 token) | 自动截断或报错 |
| 生成-4 | llama-server 不可用 | 返回 503 + 错误消息 |

### 2.3 验证模块

| 用例 | 输入 | 预期 |
|------|------|------|
| 验证-1 | 答案关键词全部在源文档中 | 通过，无警告标记 |
| 验证-2 | 答案包含源文档外的术语 | 标记"⚠️ 部分信息未经源文档确认" |
| 验证-3 | 引用 [3] 但只有 2 条检索结果 | 标记引用不完整警告 |

### 2.4 知识图谱操作

| 用例 | 输入 | 预期 |
|------|------|------|
| KG-1 | Cypher: MATCH (s:Symptom) RETURN count(s) | 返回实体数 ≥ 赛题数据的 80% |
| KG-2 | MERGE 已存在实体 | 幂等，不重复创建 |
| KG-3 | 导入后检查孤立节点 | 孤立节点数 = 0 |
| KG-4 | 导入后检查 Source 属性 | 所有节点 source_doc 非空 |

### 2.5 反馈与优化

| 用例 | 输入 | 预期 |
|------|------|------|
| 反馈-1 | rating=1 的反馈 | 存入 SQLite，点赞数 +1 |
| 反馈-2 | 30 条反馈后触发优化 | 新 α/β/γ ≠ 默认值 |
| 反馈-3 | 参数越界 α=-0.1 | 自动钳制到 [0, 1] |

### 2.6 知识抽取

| 用例 | 输入 | 预期 |
|------|------|------|
| 抽取-1 | 规范文本"故障现象：发动机漏油" | 正则命中，提取到三元组 |
| 抽取-2 | 不规范文本"发动机最近老是漏油" | 正则未命中 → 调用 1.5B few-shot |
| 抽取-3 | 上传损坏的 PDF | 返回文件解析错误 |

## 三、集成测试（模块协作）

| 用例 | 流程 | 验证点 |
|------|------|--------|
| 集成-1: 端到端问答 | POST /ask "接触器线圈烧毁" → 检索 → 生成 → 返回 | 答案包含原因和维修步骤，有引用 |
| 集成-2: 流式输出 | POST /ask → SSE 流 | 第一条数据 <3 秒到达 |
| 集成-3: 图+向量融合 | 问题使用同义词，图检索命中但 BM25 不命中 | 向量检索补齐，融合结果 ≥3 条 |
| 集成-4: 知识抽取→入库 | POST /parse → POST /confirm → GET /symptoms | 新故障现象出现在联想列表 |
| 集成-5: 反馈→调优 | 提交 30 条点赞 → 触发优化 → 检查 params.json | α/β/γ 有变化 |
| 集成-6: 导出→导入 | /export → 换设备 → /import → 查实体数 | 实体数 ≥ 导出时的 95% |
| 集成-7: 语音→问答 | POST /transcribe → 文本填入 → POST /ask | 语音识别误差 <15%，问答结果正常 |
| 集成-8: 多轮对话 | 第一轮问"燃油泵漏油"，第二轮"怎么修" | 第二轮答案引用第一轮的实体 |
| 集成-9: 图谱可视化 | GET /symptoms → 前端渲染 | 力导向图正常绘制，节点可点击展开 |
| 集成-10: 答案导出 | POST /ask → GET /report | Word 文件包含问题和引用来源 |

## 四、验收测试（赛题评分要点）

### 4.1 性能测试

| 指标 | 测试方法 | 通过标准 |
|------|----------|----------|
| 响应时间 | 50 次问答取 P95 | ≤3 秒 |
| 内存占用 | 任务管理器监控全程 | ≤8 GB |
| 启动时间 | 双击 exe → 窗口出现 | ≤30 秒 |
| 离线运行 | 拔网线后测试全部功能 | 正常 |

### 4.2 知识覆盖测试

| 指标 | 测试方法 | 通过标准 |
|------|----------|----------|
| 实体数 | `MATCH (n) RETURN count(n)` | ≥500 |
| 关系数 | `MATCH ()-[r]->() RETURN count(r)` | ≥80 |
| 设备覆盖 | 查至少 3 类设备都有故障数据 | 发动机/液压/电气各 ≥10 条 |

### 4.3 准确率测试

准备 **50 道标准测试题**，每题标注"标准答案关键词"（至少 3 个关键词）：

```
示例测试题：
Q1: "接触器线圈烧毁的原因是什么？"
标准关键词: ["电压过低", "线圈短路", "机械卡阻"]
→ 系统回答中包含 ≥2 个关键词 → 本题通过
```

| 指标 | 方法 | 通过标准 |
|------|------|----------|
| Top1 精确匹配 | 标准答案关键词全中 | ≥75%（50 题中 ≥38 题） |
| 召回率 | 专家逐条评估答案质量 | ≥90% |
| 可用性 | 答案是否可用（即使不完美） | ≥80% |

### 4.4 交互测试

| 测试项 | 验证方法 |
|--------|----------|
| 文本输入 | 输入自然语言问题，正常回答 |
| 关键词检索 | 输入"接触器 烧毁"，返回相关结果 |
| 流式显示 | 答案逐字出现，不是一次性显示 |
| 语音输入 | 说话 → 转文字 → 自动填入 → 发送 |
| 引用溯源 | 鼠标悬浮 [1]，弹出来源信息 |
| 图谱可视化 | 力导向图可拖拽缩放，点击展开 |
| 历史对话 | 之前的问题出现在历史列表 |

### 4.5 可靠性测试

| 测试项 | 验证方法 |
|--------|----------|
| 防幻觉 | 问知识库外的问题（如"核反应堆怎么修"），答案应说"未找到" |
| 低可信警告 | 生成内容中混入源文档不存在的词，验证警告标记出现 |
| 引用完整性 | 答案中的 [N] 全部在检索结果范围内 |
| 断网可用 | 拔网线，所有功能正常 |

## 五、测试数据准备

### 5.1 测试用问题集（50 题）

按设备分类，均匀覆盖：

| 类别 | 问题数 | 覆盖 |
|------|:--:|------|
| 接触器 | 8 | 线圈烧毁、触头熔焊、无法吸合等 |
| 电机 | 10 | 不能起动、过热、振动异常等 |

| 电站设备 | 8 | 发电机、配电装置故障 |
| 报警装置 | 6 | 火灾报警、监测系统故障 |
| 边缘场景 | 10 | 口语化描述、同义词、知识库外问题 |

### 5.2 测试用文档

准备 2-3 份 Word/PDF 测试文档用于验证知识抽取功能。

## 六、测试时机与责任人

| 阶段 | 测试类型 | 时机 | 测试数据 |
|------|----------|------|----------|
| 开发中 | 单元测试 | 每写完一个函数就跑 | 构造的典型输入 |
| 模块完成 | 集成测试 | 模块之间联调时 | 模拟的用户问答 |
| 全部完成 | 验收测试 | 打包前最后一周 | 50 题标准测试集 |
| 打包后 | 部署测试 | 生成 exe 后 | 纯净 Windows 虚拟机 |
