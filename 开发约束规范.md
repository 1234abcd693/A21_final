# 船舶装备故障诊断智能问答系统 —— 开发约束规范

> 依据：GB/T 8567《计算机软件文档编制规范》+ 赛题技术要求

---

## 一、编码约束

### 1.1 Python 硬性规则

```python
# ❌ 禁止
def q(x): ...                  # 单字母变量名
password = "123456"            # 硬编码密码
session = driver.session()     # 每次请求新 session，未关闭

# ✅ 必须
def search_symptoms(keyword: str) -> list[dict]:
    ...

NEO4J_PASSWORD = os.getenv("NEO4J_PASSWORD")

with driver.session() as session:  # 自动关闭
    result = session.run("...", keyword=keyword)
```

**命名规则（强制）**：函数 `snake_case` 动词开头，类 `PascalCase`，常量 `UPPER_SNAKE_CASE`，私有函数 `_` 前缀。

**类型注解（强制）**：所有函数入参和返回值必须写类型注解。

### 1.2 Cypher 硬性规则

```cypher
// ❌ 禁止：无 LIMIT、字符串拼接
MATCH (n) RETURN n
"MATCH (s:Symptom {name: '" + user_input + "'})"

// ✅ 必须：参数化 + LIMIT
MATCH (s:Symptom {name: $name})-[r:CAUSED_BY]->(c:Cause)
RETURN c.name, r.priority ORDER BY r.priority LIMIT $limit
```

**写入规则**：用 `MERGE`，禁止 `CREATE`；写入前 Browser 验证；写入后查询验证；每次写入前 Dump 备份。

## 二、接口契约

所有接口必须先在 `api-contract.md` 中定义，双方确认后才能实现。变更流程：改文档→通知→确认→改代码。禁止先改代码后补文档。

---

## 三、测试规范

| 级别 | 内容 | 最低覆盖率 |
|------|------|-----------|
| 单元测试 | 每个 API 函数、每个 Cypher 查询模式 | 80% |
| 集成测试 | API → Neo4j 完整链路 | /health /symptoms /ask 三个必须 |
| 问答准确率 | 50 条测试问题 | Top1 ≥ 75% |

---

## 四、版本控制（Git）

分支策略：`main → dev → feat/xxx`，禁止直交 main/dev。提交信息格式：`feat: 描述` 或 `fix: 描述`。合并前 `git pull --rebase origin dev`，冲突面谈解决。

---

## 五、质量保证

### 知识图谱数据质量

| 检查项 | 方法 | 频率 |
|--------|------|------|
| 孤立节点 | `MATCH (n) WHERE NOT (n)--() RETURN labels(n), count(n)` | 每次导入后 |
| Cause 无 FIXED_BY | `MATCH (c:Cause) WHERE NOT (c)-[:FIXED_BY]->() RETURN c.name` | 每次导入后 |
| 优先级重复 | 查同名 Cause 同 Symptom 下 priority 重复 | 每周 |
| Source 缺失 | `MATCH (n) WHERE n.source_page IS NULL RETURN labels(n), count(n)` | 每次导入后 |

### 代码审查

每次合并到 dev 前对方审查：类型注解完整、异常处理、参数化查询、无硬编码密码、API 格式与契约一致。

---

## 六、变更控制

知识图谱变更：B 改设计文档→Browser 试跑→通知 A→确认→导入→质量检查。

API 变更：A 改 api-contract.md→通知 B→确认→改代码→集成测试。
